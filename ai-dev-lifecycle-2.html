Establish the single source of truth. Get the README right before a single line of code is written.

The Minimal README

README.md
  • —Application name & overview one paragraph: what it is and who it's for

  • —Architecture modular patterns, data isolation strategies, security layers

Writing the PRD

.spec/<feature-name>.md
  • —User stories "As a [user], I want [action] so that [outcome]" · each has passes: false until verified
  • —Non-functional requirements measurable latency, throughput, error rate, security, and availability targets
  • —API contract REST resource hierarchy & verbs, or gRPC .proto definitions; pagination & versioning strategy
  • —Data model entities, indexes, soft-delete patterns, schema evolution, storage engine boundaries
  • —Out of scope explicit exclusions prevent scope creep inside the autonomous loop
Execute
08 / 15

Three Areas That Break Everything

Consistently under-specified — the biggest source of downstream problems

NFR

Non-Functional Requirements

Measurable targets, not vague directives. "p99 under 200ms at 500 RPS" — not "make it fast."

API

API Contract Design

Deliberate protocol choice with clear conventions. Lock in resource hierarchy and versioning before the loop runs.

DB

Data Modeling

Schema decisions are hard to change mid-flight. Normalization, indexing, and soft-delete patterns decided upfront.

Execute
10 / 15

What the Loop
Guarantees

  • —Resumable reads .progress.txt and continues from last checkpoint after any interruption
  • —No broken commits lint + typecheck + tests must pass before git commit
  • —Codebase memory consolidates reusable patterns into a ## Codebase Patterns section each run
  • —Safe execution runs inside Docker; --dangerously-skip-permissions is contained
Important constraints

Token limits are real. Long runs may hit rate limits — the progress log lets you resume without losing work.

"Done" must be explicit in acceptance criteria. The loop only marks passes: true when criteria are verifiably met.

Definition of done is a human responsibility — not the loop's.

Execute
11 / 15

Review & Merge

Your responsibilities as the human in the loop

  • —Verify acceptance criteria does the PR actually do what the story asked?
  • —Assess architecture are the choices consistent with the README and established patterns?
  • —Evaluate tests not just pass rate — are they testing the right things meaningfully?
  • —Update the PRD treat it as a living document reflecting all decisions made during review
  • —Merge to main short-lived branches, trunk-based development, no long-running feature branches
Phase 03
12 / 15
03
Ship Deploy Every merge to main is a potential release

Trunk-based deployment

  • —Main is the source of truth no staging branches, no long-lived release branches
  • —CI/CD picks up every merge automated pipeline, no manual deployment steps
  • —Fast iteration the loop runs again on the next feature; continuous forward progress
Outcomes

What This Solves

  • —Eliminates inconsistent AI-generated code persistent context keeps architecture and naming coherent across sessions
  • —Reduces specification overhead minimal PRD structure balances rigor with speed; no boilerplate for the sake of it
  • —Creates resumable, contained loops no work lost to rate limits or interruptions; progress always logged
  • —Enforces quality gates no broken code reaches main; tests run before every single commit
  • —Keeps humans in control review, architecture decisions, and "done" definitions remain human responsibilities
